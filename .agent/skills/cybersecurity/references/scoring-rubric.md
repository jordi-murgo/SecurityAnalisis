# Scoring Rubric

## Severity Levels (CVSS-Aligned)

| Score Range | Severity | Color | Description | Examples |
|---|---|---|---|---|
| 90-100 | CRITICAL | Red | Immediate exploitation possible, high impact | RCE, SQL injection with data access, auth bypass allowing full account takeover, container escape, hardcoded admin credentials exposed |
| 70-89 | HIGH | Orange | Exploitation likely, significant impact | Stored XSS, privilege escalation, SSRF to internal services, insecure deserialization, JWT algorithm confusion |
| 40-69 | MEDIUM | Yellow | Exploitation possible but requires conditions | Reflected XSS, CSRF on non-critical functions, information disclosure (stack traces, version info), weak cryptographic algorithm |
| 20-39 | LOW | Blue | Limited exploitability or impact | Missing security headers (non-critical), verbose error messages, minor misconfiguration, deprecated but non-vulnerable function usage |
| 0-19 | INFO | Gray | Best-practice deviation, defense-in-depth | Missing Content-Security-Policy report-only, non-security-critical missing input validation, code style issues with security implications |

## Confidence Tiers

| Level | Range | Criteria | Report As |
|---|---|---|---|
| HIGH | 90-100% | Vulnerable pattern confirmed + attacker-controlled input flows to sink + no compensating controls visible in analyzed scope | Finding (VULN-XXX) |
| MEDIUM | 60-89% | Pattern matches but framework may provide protection not visible in current scope (ORM parameterization, template auto-escaping, middleware protection) | Finding with caveat |
| LOW | 30-59% | Loosely matches known-bad pattern but strong possibility of framework mitigation or environmental protection | Potential issue (note) |
| INFO | <30% | Best-practice deviation or defense-in-depth recommendation, not a concrete vulnerability | Observation only |

### Confidence Modifiers

**Factors that INCREASE confidence (+10-30%):**
- Direct user input (request params, form data, URL path) flows to dangerous sink without visible sanitization
- No framework-level protections detected (raw SQL library, no template engine, manual HTML construction)
- Code handles sensitive data (PII, credentials, financial data, health records)
- Component is internet-facing (route handler, API endpoint, webhook receiver)
- Pattern confirmed in multiple locations (systemic issue)
- No input validation visible anywhere in the call chain

**Factors that DECREASE confidence (-10-30%):**
- Framework known to auto-protect for this vulnerability class (Django ORM → SQLi, React JSX → XSS)
- Code is behind authentication (reduces attack surface)
- Internal-only component (admin panel, internal API, background worker)
- Test code (but NOT for hardcoded real credentials)
- Read-only operation (reduces impact of injection)
- Compensating controls visible elsewhere (WAF rules, CSP headers, rate limiting)

### Minified/Generated Code Rule

When reviewing files that appear to be minified or machine-generated, apply special handling:

**Detection criteria** (any 2+ = treat as minified/generated):
- File matches `*.min.js`, `*.min.css`, `*.bundle.js`, `*.chunk.js`
- Single-line file exceeding 1KB (no meaningful whitespace/newlines)
- File header contains `// Generated by`, `/* auto-generated */`, `// DO NOT EDIT`
- Average variable name length < 3 characters (mangled: `a`, `b`, `_0x` prefixes)
- No comments, no blank lines, in a file > 500 characters

**Confidence adjustment**:
- Multiply confidence score by **0.5** (50% reduction) for any finding in minified/generated code
- Add caveat to finding: "This file appears minified/generated. Findings may be inaccurate. Review the source file instead."

**Preferred action**:
- If a source map (`.map` file) or unminified equivalent exists in the project, review that instead
- If the file is clearly build output (`dist/`, `build/`, `.next/`, `__generated__/`), skip entirely per the Large Codebase Strategy exclusion rules
- Only report findings from minified code if no source alternative is available AND confidence remains MEDIUM+ after the 0.5 multiplier

## Inline Suppression Comments

### Recognized Suppression Markers

The skill must detect these inline suppression comments used by other security tools:

| Marker | Tool | Language | Example |
|--------|------|----------|---------|
| `# nosec` | Bandit | Python | `eval(expr)  # nosec B307` |
| `# nosemgrep` | Semgrep | Any | `password = "test"  # nosemgrep` |
| `// NOLINT` / `// NOLINTNEXTLINE` | clang-tidy/cpplint | C/C++ | `gets(buf); // NOLINT` |
| `@SuppressWarnings` | Java compiler/SpotBugs | Java | `@SuppressWarnings("security")` |
| `// eslint-disable` | ESLint | JS/TS | `// eslint-disable-next-line security/detect-eval-with-expression` |
| `# noqa` | Flake8/Ruff | Python | `exec(code)  # noqa: S102` |
| `// noinspection` | JetBrains IDEs | Any | `// noinspection InsecureHashAlgorithm` |
| `#pragma warning(disable)` | MSVC | C/C++ | `#pragma warning(disable: 4996)` |
| `// nolint` | golangci-lint/gosec | Go | `// nolint:gosec` |
| `# rubocop:disable` | RuboCop | Ruby | `# rubocop:disable Security/Eval` |
| `// @ts-ignore` / `// @ts-expect-error` | TypeScript | TS | Security-relevant if suppressing type safety checks |

### Handling Rules

1. **Never skip a finding because of a suppression comment.** The suppression was written for a different tool with different detection logic. Our analysis is independent.

2. **DO note the suppression in the finding output.** Add a field:
   ```
   Suppression: `# nosec B307` present at line 42
   Note: A previous security tool's finding was suppressed here.
   Verify the suppression rationale is still valid.
   ```

3. **Increase suspicion, don't decrease it.** A suppression comment on a dangerous pattern means someone KNEW it was risky and chose to bypass the warning. Apply a confidence INCREASE of +10% (multiply by 1.1, capped at 1.0) rather than decrease — this code was already flagged by another tool.

4. **Exception for documented compensating controls**: If the suppression includes an explanatory comment describing a valid compensating control, report the finding at normal confidence (no +10%) and reference the claimed control for human verification:
   ```python
   # nosec B307 — input is validated against allowlist on line 38
   result = eval(expression)
   ```
   Output: "Suppression with justification. Claimed compensating control: input allowlist validation at line 38. Verify manually."

5. **Detection patterns per language**:
   - Python: `# ?(nosec|noqa|nosemgrep|type: ?ignore)`
   - JS/TS: `\/[\/\*] ?(eslint-disable|@ts-ignore|@ts-expect-error|nosemgrep)`
   - Java: `@Suppress(Warnings|Lint)`
   - C/C++: `\/\/ ?(NOLINT|NOLINTNEXTLINE)|#pragma warning\(disable`
   - Go: `\/\/ ?nolint`
   - Ruby: `# ?rubocop:disable`
   - General (all languages): `nosemgrep|noinspection`

## Score Calculation Formula

```
Finding Score = Base Severity (0-100, CVSS-aligned)
              × Confidence Multiplier (0.3 - 1.0)
              × Exploitability Factor (0.5 - 1.0)
              ± Context Adjustment (-20 to +20 points)
```

### Base Severity
Determined by vulnerability class. Reference CVSS v4.0 base metrics:
- Attack Vector (Network > Adjacent > Local > Physical)
- Attack Complexity (Low > High)
- Privileges Required (None > Low > High)
- User Interaction (None > Required)
- Impact on Confidentiality, Integrity, Availability (High > Low > None)

### Exploitability Factor
- 1.0: Exploit is trivial (known public exploit, no prerequisites)
- 0.8: Exploit requires moderate skill (custom payload needed)
- 0.6: Exploit requires specific conditions (timing, race condition, specific state)
- 0.5: Exploit requires chaining with another vulnerability

### Context Adjustment
- +20: Internet-facing + handles PII/financial data + no auth required
- +10: Internet-facing OR handles sensitive data
- 0: Standard internal application
- -10: Internal-only, behind VPN, or admin-only access
- -20: Test/development code only (except credential findings)

## Category Scoring (0-100 per agent)

Each of the 8 specialist agents returns a category score:

```
Category Score = 100 - (sum of weighted finding deductions)

Deductions per finding:
  CRITICAL (HIGH confidence): -25 points
  CRITICAL (MEDIUM confidence): -15 points
  HIGH (HIGH confidence): -15 points
  HIGH (MEDIUM confidence): -10 points
  MEDIUM (HIGH confidence): -8 points
  MEDIUM (MEDIUM confidence): -5 points
  LOW: -2 points
  INFO: -0 points

Minimum score: 0 (floor)
```

## Overall Score Aggregation

```
Overall Score = (Vuln_Score × 0.20) + (Auth_Score × 0.15) +
               (Secret_Score × 0.10) + (Dep_Score × 0.10) +
               (IaC_Score × 0.10) + (Threat_Score × 0.15) +
               (AI_Score × 0.10) + (Logic_Score × 0.10)
```

## Grade Interpretation

| Grade | Score | Meaning | Recommended Action |
|---|---|---|---|
| A | 90-100 | Excellent security posture | Maintain current practices, minor hardening |
| B | 75-89 | Good with minor issues | Address findings in regular sprint |
| C | 50-74 | Needs significant improvement | Prioritize security sprint, fix all HIGH+ |
| D | 25-49 | Serious security concerns | Security emergency — fix CRITICAL immediately |
| F | 0-24 | Critical — immediate action required | Stop deployment, incident response mode |

### Auto-CRITICAL Gate
If ANY finding has Severity=CRITICAL AND Confidence=HIGH:
- Overall grade is CAPPED at C regardless of score
- Finding must appear first in the report
- Warning banner displayed in Executive Summary

## Finding ID Format

```
[VULN-001] Title
[VULN-002] Title
...
[CHAIN-001] Attack Path Title (for multi-finding chains)
[VERIFY-001] Title (for findings that need manual verification)
```

Numbering is sequential across ALL agents. The orchestrator renumbers after aggregation to avoid duplicates.
